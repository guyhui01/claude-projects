BP-002 — Flux Inscription B2B (US-005)

De la saisie du formulaire à la confirmation + notification admin
US-005
👤 Thomas Mercier
⚙️ SiretConstraintValidator · hook_user_insert · AccountValidationService
🗓 Sprint 1
Flux de création de compte B2B en 3 phases : validation formulaire (SIRET + email), création user avec statut en_attente, puis double notification (Thomas + Admin Company-Test via SendGrid).
🔄 Diagramme de séquence — sequenceDiagram
sequenceDiagram
    actor Thomas as Thomas (Acheteur B2B)
    participant Form as Formulaire Drupal
    participant SiretVal as SiretConstraintValidator
    participant DB as MySQL
    participant Hook as hook_user_insert
    participant AccountSvc as AccountValidationService
    participant SendGrid as SendGrid API
    participant Admin as Admin Company-Test

    Thomas->>Form: Remplit formulaire (SIRET, email, mdp)
    Form->>SiretVal: Validation blur (SIRET)
    alt SIRET invalide (different de 14 chiffres)
        SiretVal-->>Form: ConstraintViolation
        Form-->>Thomas: Erreur inline - SIRET invalide
    else SIRET valide (14 chiffres)
        SiretVal-->>Form: OK
    end

    Thomas->>Form: Soumet le formulaire
    Form->>DB: Verifie unicite email
    alt Email deja enregistre
        DB-->>Form: Conflict - email existant
        Form-->>Thomas: Erreur - Un compte existe avec cet email
    else Email disponible
        DB-->>Form: OK
    end

    Form->>DB: INSERT user (status=en_attente, field_siret)
    DB-->>Form: user_id cree
    Form->>Hook: hook_user_insert(user)
    Hook->>AccountSvc: notifyNewRegistration(user)
    AccountSvc->>SendGrid: POST /mail/send (template confirmation-inscription, to Thomas)
    SendGrid-->>AccountSvc: 202 Accepted
    AccountSvc->>SendGrid: POST /mail/send (template alerte-admin, to Admin)
    SendGrid-->>AccountSvc: 202 Accepted
    SendGrid-->>Thomas: Email - Demande recue, validation sous 24h
    SendGrid-->>Admin: Email - Nouveau compte a valider
    Form-->>Thomas: Page confirmation (route Company-TestsRegisterController)
📋 Étapes clés
1 — FRONT
Validation SIRET au blur (pas en temps réel)
SiretConstraintValidator::validate()
2 — DÉCISION
Unicité email vérifiée avant création
user.register form validation
3 — BACK
User créé avec statut en_attente
field_compte_statut = en_attente
4 — BACK
hook_user_insert déclenché
Company-Test_b2b.module
5 — EXT
2 emails SendGrid envoyés
template: confirmation-inscription, alerte-admin
⚠️ Cas d'erreur
SIRET ≠ 14 chiffres → erreur inline blur
Email déjà utilisé → liens Se connecter + Mot de passe oublié
SendGrid down → log Drupal, email retry via queue

BP-003 — Flux Connexion B2B + Flood Control (US-006)

Machine à états : login → Flood API → B2bLoginSubscriber → statut compte → session
US-006
👤 Thomas Mercier
⚙️ user.login · Flood API · B2bLoginSubscriber
🗓 Sprint 1
4 états possibles à l'issue d'une tentative de connexion : session créée (compte actif), blocage temporaire (5 tentatives, flood_window config), compte en attente (validation admin), compte refusé. Le B2bLoginSubscriber intercepte UserLoginEvent pour vérifier le statut B2B.
🔄 Flowchart — Machine à états connexion
flowchart TD
    A([Acces /user/login]) --> B[Saisie email + mot de passe]
    B --> C{Flood API\nTentatives >= 5 ?}
    C -->|Oui| BLOCK[Compte bloque]
    BLOCK --> COUNTDOWN[Affiche countdown mm:ss\ndepuis flood_window config]
    COUNTDOWN --> DISABLED[Bouton Se connecter desactive]
    DISABLED --> FORGOT[Lien Mot de passe oublie ?]

    C -->|Non| CHECK[Drupal verifie identifiants\nMD5/bcrypt hash]
    CHECK --> VALID{Identifiants\ncorrects ?}

    VALID -->|Non| FLOOD1[Flood API +1 tentative]
    FLOOD1 --> COUNT2{Tentatives >= 2 ?}
    COUNT2 -->|Oui| SHOW_COUNT[Affiche compteur\nrestantes 3/5]
    COUNT2 -->|Non| SIMPLE_ERR[Message erreur simple\nsans compteur]
    SHOW_COUNT --> RETRY[Retour formulaire]
    SIMPLE_ERR --> RETRY

    VALID -->|Oui| FLOOD_RESET[Flood API reset\npour cet email]
    FLOOD_RESET --> SUBSCRIBER[B2bLoginSubscriber\ndeclenche UserLoginEvent]
    SUBSCRIBER --> STATUS{field_compte_statut ?}

    STATUS -->|en_attente| WAIT_KICK[Invalide session\nimmediate]
    WAIT_KICK --> WAIT_MSG[Affiche message:\nValidation sous 24h\n+ bouton Contacter]

    STATUS -->|refuse| REFUSE_KICK[Invalide session\nimmediate]
    REFUSE_KICK --> REFUSE_MSG[Affiche message:\nDemande refusee\n+ bouton Contacter]

    STATUS -->|active| SESSION[Session Drupal creee\nRoles b2b_buyer assignes]
    SESSION --> REDIRECT[Redirect /catalogue]

US-002

Le B2bPriceSubscriber intercepte PriceCalculateEvent pour chaque produit : si l'utilisateur a le rôle b2b_buyer, le prix négocié est affiché ; sinon, stopPropagation() masque le prix et affiche les CTA de conversion. Le sélecteur de variations (bobines) recalcule le prix total via Ajax.

BP-007 — Flux Paiement Stripe (US-010)

Stripe.js (front) → PaymentIntent (back) → Webhook asynchrone → statut commande
US-010
👤 Thomas Mercier
⚙️ commerce_stripe · Stripe PHP SDK · Webhook handler
🔌 Stripe API
🗓 Sprint 2
Pattern PaymentIntent de Stripe : le secret est créé côté serveur, transmis au navigateur, Stripe.js prend en charge la saisie CB de façon sécurisée (PCI DSS). La confirmation de paiement arrive via webhook HMAC (asynchrone), évitant les doublons en cas de réseau instable.
🔄 Diagramme de séquence — PaymentIntent + Webhook Stripe
sequenceDiagram
    participant Browser as Navigateur (Stripe.js)
    participant Drupal as Drupal Backend
    participant StripeSDK as Stripe PHP SDK
    participant StripeAPI as Stripe API (externe)
    participant Webhook as Webhook Handler Drupal
    participant DB as MySQL

    Browser->>Drupal: POST /checkout/etape-paiement (order_id, montant)
    Drupal->>StripeSDK: PaymentIntent::create(amount, currency=EUR, metadata)
    StripeSDK->>StripeAPI: POST /v1/payment_intents
    StripeAPI-->>StripeSDK: pi_xxx (id + client_secret)
    StripeSDK-->>Drupal: PaymentIntent object
    Drupal-->>Browser: Page Stripe Elements + client_secret (token)

    Note over Browser: Stripe.js monte formulaire CB securise (PCI DSS - hors scope Drupal)

    Browser->>StripeAPI: stripe.confirmCardPayment(client_secret, card_data)

    alt Carte necessite 3D Secure
        StripeAPI-->>Browser: Redirect 3DS (iframe banque)
        Browser-->>StripeAPI: Authentification 3DS validee
    end

    StripeAPI-->>Browser: PaymentIntent status=succeeded

    par Webhook asynchrone (fiable)
        StripeAPI->>Webhook: POST /stripe/webhook (payment_intent.succeeded)
        Webhook->>Webhook: Verifie signature HMAC Stripe-Signature
        Webhook->>DB: UPDATE order SET etat=processing, stripe_pi_id=pi_xxx, paid_at=NOW()
        Webhook-->>StripeAPI: HTTP 200 OK
    and Confirmation synchrone (UI)
        Browser->>Drupal: POST /checkout/complete (payment_intent_id)
        Drupal->>DB: Verifie order deja mis a jour par webhook
        Drupal-->>Browser: Redirect /commande/confirmation/order_id
    end
🔐 Sécurité Stripe
PCI DSS
CB jamais transmise à Drupal — Stripe.js only
Webhook HMAC
Signature vérifiée (Stripe-Signature header)
Idempotence
Webhook ≥ 1 fois → vérif avant UPDATE
